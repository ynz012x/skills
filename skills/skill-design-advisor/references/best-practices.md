# Skill 写作最佳实践

基于 Anthropic 团队使用数百个 Skills 的经验总结。在设计建议中，根据 skill 的类别和模式选择 3-5 条最相关的实践，并给出针对性的应用说明。

---

## 1. 不写显而易见的事

**原则**：Agent 对代码库和编程已有大量知识，skill 应聚焦于能打破其默认思维模式的信息。

**失败模式**：skill 充斥着 agent 已知的通用知识，真正有价值的差异化信息被淹没。

**最适用场景**：
- 类别：库与 API 参考、代码质量与审查
- 模式：Tool Wrapper

**反模式**：把官方文档原文搬进 skill。

**实践要点**：
- 重点写 agent 经常犯错的地方，而非正确用法的教学
- 测试方法：如果删掉某段内容，agent 还能自己做对，那就删掉它
- 内部库或 agent 掌握不稳的公共库才需要详细指导

---

## 2. 建设 Gotchas 清单

**原则**：skill 中信号最强、最有价值的部分是 Gotchas（易踩坑事项），围绕 agent 反复撞上的失败点积累。

**失败模式**：skill 写完后再不更新，同样的错误反复出现。

**最适用场景**：
- 类别：库与 API 参考、代码脚手架与模板、运维手册
- 模式：Tool Wrapper、Generator

**反模式**：只有正面指导，没有"不要做什么"的警告。

**实践要点**：
- 随着使用不断追加新发现的坑点，skill 是活的文档
- Gotchas 格式统一：问题描述 + 错误示例 + 正确做法
- 这是 skill 随时间变好的核心机制

---

## 3. 利用文件系统做渐进披露

**原则**：Skill 是一个文件夹，不只是 markdown 文件。把整个文件系统视为上下文工程与渐进披露的一部分。

**失败模式**：所有内容塞进 SKILL.md，导致文件过长、触发时加载过多无关信息。

**最适用场景**：
- 类别：所有类别（通用实践）
- 模式：Tool Wrapper（这就是该模式的核心理念）

**反模式**：SKILL.md 超过 500 行还不拆分。

**实践要点**：
- SKILL.md 只放流程指令和链接，详细内容拆到 `references/`
- 输出模板放 `assets/templates/`，检查清单放 `assets/checklists/`
- 代码示例和函数签名放 `references/examples/` 或 `references/api.md`
- 告诉 agent 目录里有什么文件，它会在合适时机自行读取

---

## 4. 不要过度约束

**原则**：给 agent 足够信息，同时留出根据具体情境灵活调整的空间。

**失败模式**：指令过于具体，agent 在不完全匹配的场景下生搬硬套或无法适应。

**最适用场景**：
- 类别：代码脚手架与模板、业务流程与团队自动化
- 模式：Generator、Pipeline

**反模式**：把每一步都写成刚性脚本，不允许任何偏离。

**实践要点**：
- Skill 有很强的复用性，指令要泛化到能覆盖常见变体
- 用"通常"、"优先"等弹性词汇替代"必须"、"总是"
- 定义约束的边界（什么必须做），但不要填满中间的实现细节

---

## 5. 考虑初始化需求

**原则**：有些 skill 需要先结合用户上下文完成初始化设置，把 setup 信息持久化以便复用。

**失败模式**：每次调用都重新询问已知信息，体验差且浪费上下文。

**最适用场景**：
- 类别：业务流程与团队自动化、数据获取与分析
- 模式：Inversion、Pipeline

**反模式**：把配置硬编码在 SKILL.md 中，换个项目就要改 skill。

**实践要点**：
- 把 setup 信息存放在 skill 目录的 `config.json` 中
- 首次运行时检测配置是否存在，不存在则引导用户填写
- 如果需要结构化输入（多选、枚举），指示 agent 使用 `AskUserQuestion` 工具

---

## 6. Description 用于触发，不是概述

**原则**：Description 字段是 agent 判断"这个请求有没有对应 skill"的依据，应描述触发时机而非功能摘要。

**失败模式**：Description 写成了摘要，导致相关请求无法触发 skill。

**最适用场景**：
- 类别：所有类别（通用实践）
- 模式：所有模式

**反模式**：`description: "一个用于代码审查的工具"` — 太笼统，无法区分触发场景。

**实践要点**：
- 主动列出所有可能的触发场景和用户表述方式
- 覆盖同义词和变体表达（如"审查"、"review"、"检查代码"）
- 格式参考："当你需要 X / 想要 Y / 在做 Z 时，使用此 skill"

---

## 7. 数据持久化与记忆

**原则**：通过在 skill 内部存储数据形成记忆机制，让 agent 能参考历史信息保持一致性。

**失败模式**：每次调用都是全新开始，丢失历史上下文和迭代积累。

**最适用场景**：
- 类别：业务流程与团队自动化、运维手册
- 模式：Pipeline（长流程需要状态持久化）

**反模式**：依赖上下文窗口记住所有历史信息。

**实践要点**：
- 简单场景：仅追加的文本日志或 JSON 文件
- 复杂场景：SQLite 数据库
- 数据应存放在稳定目录（如 `${CLAUDE_PLUGIN_DATA}`），不要放在 skill 目录内（升级时可能被删）
- 例：standup skill 维护 `standups.log`，下次运行时读取历史判断增量变化

---

## 8. 提供脚本，让 Agent 生成代码

**原则**：向 agent 提供脚本与函数库，让它把有限轮次花在"组合能力"上，而非重建样板代码。

**失败模式**：每次调用 agent 都从零写辅助函数，浪费 token 且结果不稳定。

**最适用场景**：
- 类别：数据获取与分析、产品验证、基础设施运维
- 模式：Tool Wrapper、Pipeline

**反模式**：把所有逻辑都写成自然语言指令，不提供可执行代码。

**实践要点**：
- 在 skill 目录中提供辅助函数库（如数据获取函数、API 封装）
- Agent 动态生成脚本组合这些积木完成高级任务
- 脚本是 skill 能给 agent 的最强大工具之一

---

## 9. 按需挂钩（On-demand Hooks）

**原则**：Skill 可以包含只在被调用时激活的 hooks，持续到 session 结束。适合强约束但不想常驻的场景。

**失败模式**：把所有安全约束都放在全局 hooks 中，导致正常开发时频繁被阻断。

**最适用场景**：
- 类别：基础设施运维、CI/CD 与部署
- 模式：Pipeline（安全关键流程）

**反模式**：安全约束要么全开要么全关，没有按场景控制的粒度。

**实践要点**：
- 典型用法：`/careful` 阻止 `rm -rf`、`DROP TABLE`、force-push 等危险操作
- 典型用法：`/freeze` 阻止编辑指定目录外的文件（调试时防止"顺手修"）
- 通过 `PreToolUse` 匹配器实现，只在需要时手动激活
