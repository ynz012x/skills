# Skill 功能类别

基于 Anthropic 内部数百个 Skills 的实践总结，Skill 按功能用途可归入 9 种类别。类别与设计模式是正交的两个维度：**类别回答"你在建什么"，模式回答"怎么组织结构"**。识别类别可以快速缩小候选模式范围。

> 类别不是互斥的。一个 skill 可能跨多个类别，此时选择最核心的作为主类别。

---

## 1. 库与 API 参考

**定义**：说明如何正确使用某个库、CLI 或 SDK，附带参考代码和易踩坑事项。

**识别信号**：用户提到"某个库的用法"、"SDK 封装"、"CLI 命令"、"内部平台工具"、"设计系统"。

**典型输入/输出**：
- 输入：用户描述使用场景或遇到的问题
- 输出：正确用法指导、代码片段、注意事项

**模式亲和**：**Tool Wrapper**（核心价值在领域知识加载）｜辅助 Generator（如果需要生成样板代码）

**易踩坑事项**：
- 不要重复 Claude 已知的公共库知识，聚焦内部库或 Claude 掌握不稳的库
- 必须有 gotchas 清单，这是此类 skill 信号最强的部分
- 参考代码放 `references/examples/`，不要塞进 SKILL.md

---

## 2. 产品验证

**定义**：描述如何测试或验证代码是否正常工作，通常配合外部工具（Playwright、tmux 等）完成。

**识别信号**：用户提到"端到端测试"、"验证流程"、"冒烟测试"、"无头浏览器"、"状态断言"。

**典型输入/输出**：
- 输入：待验证的功能/流程
- 输出：验证通过/失败报告、截图或日志

**模式亲和**：**Pipeline**（多步骤验证流程）+ **Reviewer**（检查点断言）

**易踩坑事项**：
- 每步都需要可编程的状态断言，不能只靠"看起来对"
- 考虑把验证脚本放在 skill 目录中，让 Claude 调用而非重写
- 验证类 skill 投入时间打磨，回报极高

---

## 3. 数据获取与分析

**定义**：连接数据体系与监控体系，包含凭证获取、dashboard ID、常见查询模式。

**识别信号**：用户提到"查数据"、"看 dashboard"、"事件漏斗"、"cohort 分析"、"Grafana"、"数据源"。

**典型输入/输出**：
- 输入：分析问题或查询需求
- 输出：数据报告、图表、查询结果

**模式亲和**：**Tool Wrapper**（脚本 + 领域知识）｜辅助 Generator（模板化报告输出）

**易踩坑事项**：
- 提供获取数据的辅助函数库，让 Claude 组合使用而非从零构建
- 需要明确数据源 UID、表名等"硬编码"信息
- 考虑把查询模式映射表放在 references/ 中

---

## 4. 业务流程与团队自动化

**定义**：把重复性工作流自动化为一条命令，通常是指令集合，可能依赖其他 skill 或 MCP。

**识别信号**：用户提到"standup"、"周报"、"建工单"、"自动化流程"、"Slack 通知"、"定期汇总"。

**典型输入/输出**：
- 输入：触发条件或时间点
- 输出：格式化的输出（帖子、工单、报告）

**模式亲和**：**Pipeline**（多步骤工作流）+ **Generator**（格式化模板输出）

**易踩坑事项**：
- 把历次执行结果保存到日志文件，帮助模型保持前后一致
- 考虑 config.json 存放频道/项目等初始化配置
- 指令不要写太死，留出根据上下文调整的空间

---

## 5. 代码脚手架与模板

**定义**：为代码库中的某类功能生成框架样板代码，可搭配可组合脚本。

**识别信号**：用户提到"新建项目"、"脚手架"、"模板"、"boilerplate"、"初始化"、"new migration"。

**典型输入/输出**：
- 输入：项目名称、类型、配置参数
- 输出：生成的文件/目录结构

**模式亲和**：**Generator**（模板驱动）+ **Tool Wrapper**（加载框架规范）

**易踩坑事项**：
- 模板不仅是代码，还包含自然语言要求（如命名规范、架构约定）
- 不要过度约束生成结果，模板定义骨架，Claude 补充细节
- 提供可组合的脚本作为积木

---

## 6. 代码质量与审查

**定义**：落实代码质量要求，辅助代码评审。可包含确定性脚本或工具。

**识别信号**：用户提到"代码审查"、"风格检查"、"测试规范"、"质量标准"、"adversarial review"。

**典型输入/输出**：
- 输入：待审代码或 PR
- 输出：审查报告、修改建议、评分

**模式亲和**：**Reviewer**（检查清单驱动）+ **Tool Wrapper**（加载编码规范）

**易踩坑事项**：
- 考虑作为 hooks 自动运行或放进 GitHub Action
- 可以拉起子 agent 做"新鲜视角"审查
- 聚焦 Claude 默认做得不够好的风格要求

---

## 7. CI/CD 与部署

**定义**：在代码库中完成拉取、推送与部署操作，可能引用其他 skill 收集数据。

**识别信号**：用户提到"部署"、"发布"、"CI/CD"、"cherry-pick"、"合并冲突"、"自动合并"。

**典型输入/输出**：
- 输入：要部署的服务/分支/PR
- 输出：部署状态、错误率比较、回滚操作

**模式亲和**：**Pipeline**（严格多步骤）+ **Tool Wrapper**（部署知识和脚本）

**易踩坑事项**：
- 关键操作必须有安全护栏（确认步骤、回滚机制）
- 冒烟测试作为阶段门禁
- 提供部署脚本而非让 Claude 每次重建

---

## 8. 运维手册（Runbooks）

**定义**：从症状出发（告警、错误特征），引导完成跨工具排查链路，生成结构化报告。

**识别信号**：用户提到"排查"、"告警"、"debugging"、"on-call"、"日志关联"、"根因分析"。

**典型输入/输出**：
- 输入：症状描述、request ID、告警信息
- 输出：排查报告、根因判断、修复建议

**模式亲和**：**Pipeline**（诊断链路）+ **Reviewer**（检查点验证）

**易踩坑事项**：
- 建立"症状 → 工具 → 查询模式"的映射关系
- 每步输出结构化中间结果，防止长链路信息丢失
- 考虑 on-demand hooks 限制危险操作

---

## 9. 基础设施运维

**定义**：执行日常维护与运维操作，部分操作具有破坏性，需安全护栏。

**识别信号**：用户提到"清理资源"、"依赖管理"、"成本调查"、"孤立 pods"、"运维操作"。

**典型输入/输出**：
- 输入：运维任务或异常信号
- 输出：操作报告、清理结果、成本分析

**模式亲和**：**Tool Wrapper**（领域知识 + 脚本）+ **Pipeline**（安全操作流程）

**易踩坑事项**：
- 破坏性操作必须有确认步骤和观察期
- 提供安全护栏脚本，防止误操作
- 让工程师更容易遵循最佳实践，而非增加负担
