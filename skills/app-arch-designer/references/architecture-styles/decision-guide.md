# 架构风格选择决策指南

## 概述

本文档提供架构风格选择的决策框架，帮助根据项目特征选择最合适的分层架构。三种候选架构：Clean Architecture、Hexagonal Architecture、Vertical Slice Architecture。

## 决策树

```
项目需求评估
│
├─ Q1: 领域复杂度如何？
│   ├─ 低（CRUD 为主）──────► 简单分层 或 Vertical Slice（轻量切片）
│   ├─ 中（有业务规则）────► 继续评估 Q2
│   └─ 高（复杂领域逻辑）──► 继续评估 Q2
│
├─ Q2: 外部依赖的多样性和可替换需求？
│   ├─ 高（多种入口/存储/第三方，需可替换）──► Hexagonal
│   ├─ 中（标准 Web 服务）────────────────────► 继续评估 Q3
│   └─ 低（单一入口、固定存储）────────────────► 继续评估 Q3
│
├─ Q3: 功能之间的独立性？
│   ├─ 高（功能模块独立、变更频率不同）──► Vertical Slice
│   ├─ 中（有交叉但可分离）──────────────► Clean Architecture
│   └─ 低（功能高度耦合）──────────────────► Clean Architecture
│
├─ Q4: 团队结构与经验？
│   ├─ Feature Team（按功能划分）──► Vertical Slice 更自然
│   ├─ Component Team（按层划分）──► Clean/Hexagonal 更自然
│   └─ 小团队全栈────────────────► 取决于 Q2/Q3 结果
│
└─ Q5: 未来演进方向？
    ├─ 可能拆分微服务──► Vertical Slice（天然微服务边界）
    ├─ 保持单体但要可维护──► Clean Architecture
    └─ 技术栈可能迁移──► Hexagonal（技术中立）
```

## 决策因素详解

### 因素 1：领域复杂度

| 指标 | 低 | 中 | 高 |
|------|----|----|-----|
| 业务规则数量 | < 10 | 10-50 | > 50 |
| 实体关系 | 简单一对多 | 多对多、树状 | 复杂图状、时态 |
| 不变量约束 | 几乎没有 | 部分关键约束 | 大量跨实体约束 |
| 领域事件 | 不需要 | 少量通知 | 核心协作机制 |

**影响**：高复杂度需要明确的领域层隔离（Clean/Hexagonal），低复杂度无需过度分层。

### 因素 2：外部依赖多样性

| 指标 | 低 | 中 | 高 |
|------|----|----|-----|
| 入口类型 | 仅 REST | REST + 定时任务 | REST + gRPC + MQ + CLI |
| 存储类型 | 单一 DB | DB + Cache | 多 DB + Cache + ES + S3 |
| 第三方集成 | 0-1 | 2-3 | > 3，且可能切换 |
| 可替换需求 | 无 | 偶尔 | 经常 |

**影响**：高多样性时 Hexagonal 的端口/适配器模型优势明显。

### 因素 3：功能独立性

| 指标 | 低 | 中 | 高 |
|------|----|----|-----|
| 功能间数据共享 | 大量共享 | 部分共享 | 各自独立 |
| 变更频率 | 均匀 | 略有差异 | 差异显著 |
| 部署需求 | 统一部署 | 统一部署 | 可能独立部署 |
| 团队分工 | 按层分工 | 混合 | 按功能分工 |

**影响**：高独立性时 Vertical Slice 减少跨功能影响。

### 因素 4：团队经验

| 概念 | 要求的经验 |
|------|-----------|
| Clean Architecture | 理解依赖倒置、分层职责、Use Case 模式 |
| Hexagonal | 理解端口/适配器、DI 容器、接口隔离原则 |
| Vertical Slice | 理解 CQRS、Mediator、Feature 内聚 |

### 因素 5：项目规模

| 规模 | 推荐倾向 |
|------|---------|
| 小型（< 10 功能） | 简单分层，避免过度架构 |
| 中型（10-50 功能） | 三种均可，取决于其他因素 |
| 大型（> 50 功能） | Vertical Slice 或 Hexagonal + 模块化 |

## 混合策略

实践中常见混合使用，而非纯粹选择一种：

### 策略 A：Hexagonal + Vertical Slice

```
适用：大型系统、多 BC、功能独立性高

每个 BC 内部采用 Hexagonal 结构（端口/适配器），
BC 之间按 Vertical Slice 组织（Feature 为主）。
```

### 策略 B：Clean Architecture + Vertical Slice

```
适用：中大型系统、需要清晰分层约束

核心/支撑域采用 Clean Architecture 四层结构，
通用域或简单 CRUD 模块采用轻量 Vertical Slice。
```

### 策略 C：按 BC 差异化

```
适用：不同 BC 复杂度差异大

核心域（高复杂度）→ Hexagonal Architecture
支撑域（中复杂度）→ Clean Architecture
通用域（低复杂度）→ Vertical Slice 或简单分层
```

## 快速决策矩阵

| 场景 | 推荐 | 理由 |
|------|------|------|
| 创业 MVP | Vertical Slice | 快速迭代、功能独立 |
| 企业核心系统 | Hexagonal | 长期维护、技术可替换 |
| SaaS 产品 | Clean Architecture | 清晰分层、团队协作 |
| 微服务前身 | Vertical Slice | 天然拆分边界 |
| 数据密集型 | Clean Architecture | 领域逻辑与数据访问分离 |
| 多渠道接入 | Hexagonal | 多适配器同一核心 |
| 中台/平台 | Hexagonal + 模块化 | 可替换性 + 模块独立 |
| 复杂度不均匀 | 混合策略 C | 按域差异化选型 |

## 反模式警告

### 过度架构

- 5 个 API 的小服务不需要 Hexagonal 的完整端口/适配器
- 只有 CRUD 的管理后台不需要 DDD 战术设计
- 原型验证阶段不需要分层架构

### 架构不匹配

- 高耦合功能强行 Vertical Slice → 大量重复代码
- 简单 CRUD 强行 Hexagonal → 接口层膨胀无实际价值
- 功能独立但强行 Clean Architecture 4 层 → 修改一个功能要改 4 个地方

### 团队不匹配

- 初级团队使用 Hexagonal → 端口/适配器混乱
- 不理解 CQRS 的团队使用 Vertical Slice → 切片间耦合失控
- 不理解依赖倒置的团队使用 Clean Architecture → 依赖方向混乱

## 决策输出

架构选择完成后，输出为 ADR（Architecture Decision Record）格式：

```markdown
## ADR-001: 架构风格选择

**状态**：已接受

**上下文**：
- 领域复杂度：[高/中/低]
- 外部依赖：[高/中/低]
- 功能独立性：[高/中/低]
- 团队经验：[具体描述]
- 演进方向：[具体描述]

**决策**：
采用 [架构风格]，因为 [核心理由]

**后果**：
- 正面：[列举]
- 负面：[列举]
- 风险：[列举]
```
