# 统一语言实践指南

## 概述

统一语言（Ubiquitous Language）是 DDD 的核心实践——业务和技术团队共享一套精确的术语体系，这套语言贯穿需求讨论、代码命名、文档编写的全过程。

## 为什么需要统一语言

- 消除业务人员和开发人员之间的"翻译损失"
- 让代码成为领域知识的直接表达
- 帮助发现限界上下文的边界（当术语含义发生变化时）

## 建立方法

### 1. 术语提取

从业务描述中提取核心名词（实体/概念）和动词（操作/行为）：

- 与业务专家对话，记录他们使用的术语
- 从需求文档、流程图、业务规则中提取
- 注意区分：业务术语 vs 技术术语（优先使用业务术语）

### 2. 精确定义

为每个术语给出明确、无歧义的定义：

- 一个术语只有一个含义
- 如果同一个词有多个含义 → 这是 BC 划分的信号
- 定义要业务人员也能理解

### 3. 歧义消除

当发现同名不同义时的处理方式：

| 情况 | 处理方式 |
|------|----------|
| 同一术语在不同 BC 含义不同 | 在各自 BC 内独立定义，不强求统一 |
| 同一 BC 内术语含义模糊 | 与业务专家确认，选择最精确的表述 |
| 技术术语与业务术语冲突 | 代码中使用业务术语，技术术语仅在基础设施层 |

### 4. 持续维护

- 术语表随项目演进持续更新
- 新成员入职时以术语表作为领域知识入口
- 代码审查时检查命名是否符合统一语言

## 术语表模板

```markdown
# 术语表 — {{project_name}}

## {{bounded_context_name}}

| 术语 | 定义 | 所属聚合 | 同义词/易混淆词 | 备注 |
|------|------|----------|----------------|------|
| {{term}} | {{precise_definition}} | {{aggregate}} | {{synonyms_or_confusing_terms}} | {{notes}} |
```

## 多上下文同名不同义

这是划分 BC 的强信号。记录方式：

```markdown
## 跨上下文术语对照

| 术语 | 在"销售"上下文 | 在"物流"上下文 | 在"财务"上下文 |
|------|---------------|---------------|---------------|
| 订单 | 客户的购买请求 | 需要配送的包裹 | 需要结算的交易 |
| 客户 | 购买商品的人 | 收货地址的主人 | 付款账户持有者 |
```

## 在代码中的体现

统一语言应直接反映在代码命名中：

- 类名 = 领域概念（实体/值对象/聚合根）
- 方法名 = 领域操作（业务动词）
- 模块/包名 = 限界上下文名称
- 避免使用 Manager、Handler、Helper 等技术泛称

```
好的命名（业务语言）：
  Order.place()
  Payment.confirm()
  Inventory.reserve(quantity)

不好的命名（技术语言）：
  OrderManager.process()
  PaymentHandler.handle()
  InventoryHelper.update()
```
