# DDD 战术设计指南

## 概述

战术设计负责将领域模型映射为代码——在限界上下文内部设计聚合、实体、值对象、领域事件等构建块。

## 聚合设计（Aggregate）

聚合是一组紧密关联的对象集合，有统一的事务一致性边界。外部只能通过聚合根（Aggregate Root）访问和修改聚合内部状态。

### Vaughn Vernon 聚合设计四规则

| 规则 | 要点 | 实践指导 |
|------|------|----------|
| **Rule 1: 在一致性边界内建模真正的不变量** | 聚合边界应包裹必须同时保持一致的数据 | 如果两个数据变更必须在同一事务中完成，它们属于同一聚合 |
| **Rule 2: 设计小聚合** | 优先设计小聚合 | 大聚合导致并发冲突、内存消耗、事务锁竞争。一个聚合可以只包含聚合根+少量值对象 |
| **Rule 3: 通过 ID 引用其他聚合** | 聚合间不持有对象引用，只保存对方 ID | 降低耦合，允许独立加载和独立扩展 |
| **Rule 4: 边界外使用最终一致性** | 聚合内强一致性，聚合间最终一致性 | 通过领域事件驱动跨聚合的状态同步 |

### 聚合设计步骤

1. 从 BC 内的业务规则出发，识别必须同时保持一致的数据组
2. 为每组确定聚合根（整个聚合的唯一入口和标识）
3. 将相关实体和值对象组织到聚合内部
4. 验证聚合大小：宜小不宜大
5. 确认跨聚合交互只通过 ID 引用 + 领域事件

### 聚合根判断标准

- 具有全局唯一标识符
- 有独立的生命周期
- 是外部访问聚合内部的唯一入口
- 负责维护聚合内所有不变量

## 实体（Entity）

### 定义

具有唯一标识的领域对象，即使所有属性值改变，只要标识相同就是同一实体。

### 特征

- 有唯一标识符（ID）
- 有生命周期（创建、修改、可能删除）
- 状态可变
- 通过 ID 区分而非属性值

### 设计要点

- 将业务行为封装在实体方法中（避免贫血模型）
- 状态变更通过方法调用，方法内校验不变量
- 不要暴露 setter，用有业务含义的方法名

## 值对象（Value Object）

### 定义

没有唯一标识的不可变对象，以属性值的组合来判定相等。

### 特征

- 无唯一标识
- 不可变（创建后不能修改）
- 以值相等（所有属性相同则相等）
- 可以自由替换

### 实体 vs 值对象判断标准

| 问题 | 实体 | 值对象 |
|------|------|--------|
| 需要跟踪它的变化历史吗？ | 是 | 否 |
| 两个属性完全相同的对象是同一个吗？ | 不一定（看 ID） | 是 |
| 它有独立的生命周期吗？ | 是 | 否（依附于实体） |
| 替换它时需要保留同一性吗？ | 是 | 否（直接替换新实例） |

### 常见值对象示例

- 金额（Money）：数值 + 币种
- 地址（Address）：省/市/区/街道/邮编
- 时间范围（DateRange）：开始时间 + 结束时间
- 坐标（Coordinate）：经度 + 纬度

## 领域服务（Domain Service）

### 何时使用领域服务

当某个业务操作：
- 不自然地属于任何单一实体或值对象
- 涉及多个聚合的协调
- 是一个无状态的领域操作

### 领域服务 vs 应用服务

| 维度 | 领域服务 | 应用服务 |
|------|----------|----------|
| 所在层 | 领域层 | 应用层 |
| 职责 | 纯业务逻辑 | 用例编排（协调领域对象 + 基础设施） |
| 状态 | 无状态 | 无状态 |
| 依赖 | 只依赖领域对象 | 依赖领域层 + 基础设施接口 |

## 领域事件（Domain Event）

### 定义

领域中已经发生的重要事实的记录。事件是不可变的，描述的是过去时（"已发生"）。

### 命名规范

使用过去时 + 领域动词：`OrderPlaced`、`PaymentConfirmed`、`InventoryReserved`

### 事件设计要素

| 要素 | 说明 |
|------|------|
| 事件名称 | 过去时动词短语 |
| 触发者 | 哪个聚合/操作产生此事件 |
| 携带数据 | 事件发生时的关键上下文信息 |
| 发生时间 | 事件时间戳 |
| 订阅者 | 哪些组件关心此事件 |

### 事件发布策略

| 策略 | 适用场景 | 一致性保证 |
|------|----------|-----------|
| 进程内同步 | 同一聚合内的副作用 | 强一致性 |
| 进程内异步 | 同一服务内的跨聚合协调 | 最终一致性 |
| 消息队列 | 跨服务/跨 BC 的事件通知 | 最终一致性 |

## 仓储（Repository）

### 设计原则

- 每个聚合根一个 Repository 接口
- Repository 接口定义在领域层
- Repository 实现在基础设施层（依赖倒置）
- 只提供聚合根级别的持久化操作

### 典型接口方法

```
Repository<AggregateRoot>:
  - save(aggregate)           # 保存聚合
  - find_by_id(id)           # 按 ID 查找
  - delete(aggregate)         # 删除聚合
  - find_by_<criteria>(...)  # 按业务条件查询
```

## 工厂（Factory）

### 何时使用

- 对象创建逻辑复杂（涉及多个步骤或条件）
- 需要确保创建出的对象满足所有不变量
- 创建逻辑不适合放在构造函数中

### 实现方式

- 聚合根上的工厂方法（最常用）
- 独立的工厂类（创建逻辑特别复杂时）
- 抽象工厂（需要创建一族相关对象时）
