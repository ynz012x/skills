# Step 1 检查清单：上下文理解

在进入 Step 2 之前，确认以下产出完整：

## 必须项

- [ ] 项目名称和背景描述已明确
- [ ] 核心业务能力清单已识别（至少 3 项主要业务能力）
- [ ] 系统利益相关者已列出（至少：产品、开发、运维）
- [ ] 质量属性优先级已排列（至少 Top 3）
- [ ] 技术约束已记录（语言、框架、数据库、基础设施）
- [ ] 组织约束已记录（团队规模、技能水平、交付节奏）
- [ ] 领域复杂度已评估（高/中/低 + 依据）

## standard/comprehensive 额外项

- [ ] 业务流程高层概述（核心流程 3-5 个）
- [ ] 非功能需求量化指标（性能、可用性、安全）
- [ ] 现有系统集成清单（如有）
- [ ] 用户角色与核心用例概述

## 完整性验证

确认以下问题已有答案：
1. 这个系统要解决什么业务问题？
2. 谁是主要用户？有哪些核心用例？
3. 有哪些技术和组织约束限制了架构选择？
4. 最重要的质量属性是什么？（可维护性 > 性能 > 可用性？）
5. 这是全新系统还是在现有系统上演进？
