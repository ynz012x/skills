# 功能需求主线

功能需求回答"做什么"的问题，描述系统应该提供的功能和行为。

## 核心理念

功能需求是系统与用户交互的具体描述，关注：
- 系统应该做什么？
- 用户如何与系统交互？
- 系统如何响应各种情况？

## 功能分类框架

### 三层功能结构

```
┌─────────────────────────────────────┐
│           业务支持功能               │  核心业务流程
│         （Business Support）         │  60-70%
├─────────────────────────────────────┤
│           管理支持功能               │  运营管理、报表
│        （Management Support）        │  20-30%
├─────────────────────────────────────┤
│           维护支持功能               │  系统配置、权限
│       （Maintenance Support）        │  10-15%
└─────────────────────────────────────┘
```

### 业务支持功能

**定义**：直接支持核心业务流程的功能

**特征**：
- 体现业务价值
- 用户日常高频使用
- 与业务流程直接相关

**示例**（电商系统）：
- 商品浏览和搜索
- 购物车管理
- 订单创建和支付
- 物流跟踪
- 售后处理

### 管理支持功能

**定义**：支持运营管理和决策的功能

**特征**：
- 面向管理人员
- 提供数据分析和报表
- 支持业务监控和审批

**示例**（电商系统）：
- 销售数据报表
- 订单审批
- 库存预警
- 客户分析
- 运营仪表盘

### 维护支持功能

**定义**：支持系统运行和维护的功能

**特征**：
- 面向管理员
- 配置和权限管理
- 系统监控和日志

**示例**（电商系统）：
- 用户权限管理
- 系统参数配置
- 操作日志查询
- 数据备份恢复
- 系统监控告警

```
├── 修改订单
│   ├── 修改收货地址
│   └── 修改配送方式
└── 取消订单
    ├── 用户取消
    └── 系统自动取消
```

### 用户故事

适用于敏捷开发：

```
作为 <角色>，
我想要 <功能>，
以便 <价值>。

验收标准：
- 条件1
- 条件2
```

**示例**：
```
作为 买家，
我想要 在订单详情页取消未支付的订单，
以便 在改变主意时能够释放库存。

验收标准：
- 只有未支付状态的订单可以取消
- 取消后库存自动恢复
- 取消后发送通知给买家
```

## 功能细化层次

### 概要级

用于整体规划和沟通：
- 功能名称
- 简要描述
- 主要参与者
- 优先级

### 详细级

用于开发实现：
- 完整用例描述
- 业务规则
- 界面原型
- 数据要求

### 细化时机

| 分析深度 | 概要级 | 详细级 |
|----------|--------|--------|
| quick | 全部功能 | 核心功能（3-5个） |
| standard | 全部功能 | 核心+重要功能（10-15个） |
| comprehensive | 全部功能 | 全部功能 |

## 功能完整性检查

### CRUD检查

对每个核心实体检查是否覆盖：
- **C**reate：创建
- **R**ead：查询
- **U**pdate：更新
- **D**elete：删除

### 生命周期检查

检查业务对象的完整生命周期：

```
创建 → 处理中 → 完成/取消 → 归档
```

每个状态转换是否有对应的功能？

### 角色覆盖检查

检查每个角色是否有足够的功能支持：

| 角色 | 业务支持 | 管理支持 | 维护支持 |
|------|----------|----------|----------|
| 普通用户 | ✓ | - | - |
| 运营人员 | ✓ | ✓ | - |
| 管理员 | - | ✓ | ✓ |
