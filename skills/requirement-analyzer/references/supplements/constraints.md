# 约束分析

约束是对项目或系统的限制条件，需求分析必须在约束范围内进行。

## 约束定义

**约束**：项目或系统必须遵守的限制条件，不能通过谈判改变，只能接受或放弃项目。

**约束 vs 需求**：
- 需求：希望系统具备的能力（可协商）
- 约束：必须遵守的限制（不可协商）

---

## 约束分类

### 技术约束

与技术选型和环境相关的限制。

| 类型 | 说明 | 示例 |
|------|------|------|
| **平台约束** | 运行平台要求 | 必须部署在公司私有云 |
| **语言约束** | 编程语言要求 | 后端必须使用Java |
| **框架约束** | 技术框架要求 | 前端必须使用React |
| **数据库约束** | 数据库选型要求 | 必须使用Oracle数据库 |
| **集成约束** | 系统集成要求 | 必须与SAP系统对接 |
| **标准约束** | 技术标准要求 | API必须符合RESTful规范 |

### 组织约束

与组织资源和管理相关的限制。

| 类型 | 说明 | 示例 |
|------|------|------|
| **预算约束** | 项目预算限制 | 总预算不超过100万 |
| **时间约束** | 项目时间限制 | 必须在12月31日前上线 |
| **人力约束** | 团队规模限制 | 开发团队最多5人 |
| **技能约束** | 团队技能限制 | 团队不熟悉Golang |
| **流程约束** | 组织流程要求 | 必须通过安全评审 |

### 法规约束

与法律法规和合规相关的限制。

| 类型 | 说明 | 示例 |
|------|------|------|
| **数据保护** | 个人信息保护 | 遵守《个人信息保护法》 |
| **行业监管** | 行业合规要求 | 金融系统需满足等保三级 |
| **知识产权** | IP保护要求 | 不能使用GPL开源组件 |
| **出口管制** | 跨境数据限制 | 数据不能出境存储 |
| **审计要求** | 审计合规要求 | 操作日志保留7年 |

### 业务约束

与业务运营相关的限制。

| 类型 | 说明 | 示例 |
|------|------|------|
| **兼容约束** | 现有系统兼容 | 不能影响现有订单处理 |
| **迁移约束** | 数据迁移要求 | 历史数据必须完整迁移 |
| **运营约束** | 运营条件限制 | 7×24小时运营，不能停机 |
| **用户约束** | 用户环境限制 | 用户使用IE11浏览器 |

## 约束识别

### 识别来源

1. **项目章程**：项目启动文件中的限制
2. **合同条款**：商务合同中的约束
3. **组织政策**：公司IT政策和标准
4. **法规研究**：适用的法律法规
5. **干系人访谈**：了解隐性约束
6. **技术评估**：现有技术环境限制

### 识别问题清单

**技术约束**：
- 有指定的技术栈或平台吗？
- 需要与哪些现有系统集成？
- 有技术标准或规范要求吗？
- 部署环境有什么限制？

**组织约束**：
- 项目预算是多少？
- 项目截止日期是什么？
- 团队规模和技能情况？
- 有什么必须遵守的流程？

**法规约束**：
- 涉及哪些法律法规？
- 有行业合规要求吗？
- 数据存储和处理有限制吗？
- 有审计或备案要求吗？

**业务约束**：
- 对现有业务有什么影响？
- 数据迁移有什么要求？
- 系统切换有什么限制？
- 用户环境有什么特殊情况？

```
约束ID：CON-001
约束名称：部署平台约束
类型：技术约束
描述：系统必须部署在公司私有云平台（华为云Stack）
来源：《IT基础设施管理规定》
影响：
- 不能使用公有云特有服务
- 需要与私有云团队协调资源
可协商性：不可协商
验证方式：部署验收测试
```

## 约束影响分析

### 约束对架构的影响

| 约束类型 | 可能的架构影响 |
|----------|----------------|
| 必须使用特定数据库 | 影响ORM选择、SQL写法 |
| 必须支持高并发 | 需要分布式架构、缓存策略 |
| 数据不能出境 | 需要本地化部署方案 |
| 必须7×24运营 | 需要高可用架构、灰度发布 |
| 预算有限 | 优先选择开源方案 |

### 约束冲突处理

当约束之间存在冲突时：

1. **识别冲突**：明确哪些约束相互矛盾
2. **评估影响**：分析各约束的重要性
3. **升级决策**：向有权限的人员报告
4. **记录决策**：文档化最终决定

**示例**：
```
冲突：预算约束 vs 安全约束
- 预算约束：总预算100万
- 安全约束：必须达到等保三级

分析：
- 等保三级建设预计需要50万
- 留给功能开发的预算只有50万

决策：
- 分两期实施
- 一期：核心功能+等保基础要求
- 二期：完整等保三级认证

决策人：项目发起人
决策日期：2024-01-15
```

## 约束与假设

### 约束 vs 假设

| 类型 | 定义 | 管理方式 |
|------|------|----------|
| **约束** | 确定的限制 | 必须遵守 |
| **假设** | 不确定但假定为真 | 需要验证 |

### 假设管理

假设如果不成立，可能变成约束或风险：

```
假设ID：ASM-001
假设内容：用户浏览器都支持ES6
验证方式：分析用户浏览器统计数据
验证日期：2024-01-20
验证结果：95%用户支持，5%使用IE11
处理措施：增加IE11兼容作为约束
```
