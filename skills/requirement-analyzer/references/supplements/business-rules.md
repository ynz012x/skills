# 业务规则分析

业务规则是组织运营中必须遵守的规定、政策和逻辑，是需求分析的重要补充。

## 业务规则定义

**业务规则**：对业务行为的约束或指导，定义了什么可以做、什么不可以做、以及如何做。

**特征**：
- 来源于业务政策、法规或行业惯例
- 独立于具体功能，可跨用例复用
- 可能随时间或政策变化而调整

---

## 业务规则分类

### 按类型分类

| 类型 | 定义 | 示例 |
|------|------|------|
| **计算规则** | 定义如何计算值 | 订单总额 = Σ(单价 × 数量) - 折扣 |
| **验证规则** | 定义数据有效性条件 | 手机号必须是11位数字 |
| **推理规则** | 定义条件推导结论 | 如果会员等级=金卡，则享受9折 |
| **约束规则** | 定义业务限制 | 单笔订单金额不超过10万元 |
| **触发规则** | 定义条件触发动作 | 库存<安全库存时，自动发送预警 |

### 按来源分类

| 来源 | 说明 | 示例 |
|------|------|------|
| **法规要求** | 法律法规强制要求 | 实名认证、数据保留期限 |
| **行业标准** | 行业通用规范 | 金融交易规则、医疗处方规范 |
| **组织政策** | 企业内部政策 | 审批权限、折扣策略 |
| **业务逻辑** | 业务运营规则 | 定价规则、积分规则 |

## 业务规则识别

### 识别来源

1. **用例分析**：从用例的前置条件、后置条件、业务流程中提取
2. **干系人访谈**：询问"有什么规定或限制？"
3. **现有文档**：查阅政策文件、操作手册、合同条款
4. **法规研究**：了解相关法律法规要求
5. **异常场景**：分析什么情况下不允许操作

### 识别问题清单

- 这个操作有什么前提条件？
- 什么情况下不允许执行？
- 数据有什么格式或范围要求？
- 金额/数量是怎么计算的？
- 有什么自动触发的逻辑？
- 不同角色/等级有什么区别？

## 业务规则描述

### 自然语言描述

适用于简单规则：

```
规则BR-001：订单取消时限规则
描述：用户只能在订单创建后24小时内取消未支付的订单
```

### 结构化描述

适用于复杂规则：

```
规则ID：BR-002
规则名称：会员折扣规则

条件：
- 用户类型 = 会员
- 会员等级 IN (银卡, 金卡, 钻石卡)

规则逻辑：
IF 会员等级 = 银卡 THEN 折扣 = 0.95
IF 会员等级 = 金卡 THEN 折扣 = 0.90
IF 会员等级 = 钻石卡 THEN 折扣 = 0.85

适用范围：
- 用例：创建订单、计算价格
- 实体：订单、订单明细

来源：《会员管理制度》v2.0
优先级：高
生效日期：2024-01-01
```

### 决策表

适用于多条件组合：

| 条件/动作 | 规则1 | 规则2 | 规则3 | 规则4 |
|-----------|-------|-------|-------|-------|
| 订单金额>1000 | Y | Y | N | N |
| 会员等级=金卡 | Y | N | Y | N |
| **折扣率** | 0.85 | 0.90 | 0.90 | 0.95 |

### 计算公式

适用于计算规则：

```
规则BR-003：运费计算规则

公式：
运费 = 基础运费 + 超重运费

其中：
- 基础运费 =
    IF 订单金额 >= 99 THEN 0
    ELSE IF 配送区域 = 市内 THEN 5
    ELSE 10

- 超重运费 =
    IF 重量 <= 1kg THEN 0
    ELSE (重量 - 1) × 2
```

## 业务规则管理

### 规则编号规范

```
BR-[模块代码]-[序号]

示例：
BR-ORD-001  订单模块第1条规则
BR-PAY-002  支付模块第2条规则
```

### 规则属性

| 属性 | 说明 |
|------|------|
| **规则ID** | 唯一标识 |
| **规则名称** | 简短描述性名称 |
| **类型** | 计算/验证/推理/约束/触发 |
| **描述** | 详细说明 |
| **条件** | 触发条件 |
| **动作** | 执行的动作或返回的结果 |
| **来源** | 规则依据 |
| **适用范围** | 关联的用例或实体 |
| **优先级** | 规则冲突时的处理优先级 |
| **生效日期** | 规则开始生效的时间 |
| **负责人** | 规则的业务负责人 |

### 规则变更管理

业务规则可能变化，需要：
1. 记录规则版本历史
2. 标记规则生效日期
3. 考虑规则可配置性
4. 评估变更影响范围

## 规则与系统设计

### 规则实现方式

| 方式 | 适用场景 | 优缺点 |
|------|----------|--------|
| **硬编码** | 稳定不变的规则 | 简单但难修改 |
| **配置表** | 参数可调的规则 | 灵活但有限 |
| **规则引擎** | 复杂多变的规则 | 强大但复杂 |

### 规则引擎适用场景

- 规则数量多（>50条）
- 规则频繁变化
- 需要非技术人员维护规则
- 规则逻辑复杂

## 业务规则文档模板

```
# 业务规则清单

## 订单模块

### BR-ORD-001：订单金额上限规则
- 类型：约束规则
- 条件：创建订单时
- 规则：单笔订单金额不得超过100,000元
- 异常处理：提示"订单金额超过上限，请拆分订单"
- 来源：财务管理制度
- 优先级：高

### BR-ORD-002：订单取消规则
- 类型：约束规则
- 条件：用户请求取消订单
- 规则：
  - 未支付订单：24小时内可取消
  - 已支付未发货：可取消，自动退款
  - 已发货：不可取消，需走售后流程
- 来源：客服管理制度
- 优先级：高
```
