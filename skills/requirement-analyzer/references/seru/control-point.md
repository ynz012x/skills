# 管控点分析（SERU-R）

管控点（Control Point）是业务流程中需要监控、控制或审核的关键节点。

## 核心概念

**管控点的定义**：
管控点是业务流程中需要进行检查、审批、监控或干预的节点，用于确保业务合规、防范风险、提升效率。

**管控点的作用**：
- 风险控制：在关键节点设置检查，防止错误传递
- 合规保障：满足内控和外部监管要求
- 效率提升：通过监控发现瓶颈，优化流程
- 决策支持：提供关键指标，辅助管理决策

## 管控点类型

| 类型 | 定义 | 示例 |
|------|------|------|
| **审批点** | 需要人工审批才能继续的节点 | 大额订单审批、折扣审批、退款审批 |
| **检查点** | 系统自动进行规则检查的节点 | 库存检查、信用检查、合规检查 |
| **监控点** | 持续监控指标或状态的节点 | 库存预警、超时监控、异常检测 |
| **记录点** | 需要留存证据或日志的节点 | 操作日志、审计轨迹、电子签章 |

## 管控点识别方法

### 从业务流程识别

在业务流程的以下位置寻找管控点：
- **输入输出边界**：外部数据进入或离开系统的点
- **状态转换点**：业务对象状态发生重大变化的点
- **金额阈值点**：涉及金额超过一定阈值的点
- **权限边界点**：需要更高权限才能执行的点

### 从风险角度识别

| 风险类型 | 对应管控点 |
|----------|------------|
| 财务风险 | 大额审批、预算控制、费用报销审批 |
| 合规风险 | 合同审批、资质审核、数据脱敏 |
| 运营风险 | 库存预警、订单超时、服务降级 |
| 安全风险 | 登录验证、权限校验、敏感操作确认 |

### 从监管要求识别

- 财务审计要求
- 行业监管规定
- 数据保护法规
- 内部控制规范

## 管控点描述模板

```
管控点名称：XXX审批/检查/监控

类型：审批点 / 检查点 / 监控点 / 记录点

所在流程：XXX流程

触发条件：
- 条件1（如：订单金额 > 10000元）
- 条件2

管控规则：
- 规则1
- 规则2

处理方式：
- 审批点：审批角色、审批时限
- 检查点：检查通过/拒绝的后续处理
- 监控点：预警阈值、通知方式
- 记录点：记录内容、保存期限

责任人：
- 角色/岗位

相关报表：
- 报表1：用于XXX监控
```

## 管控点设计原则

### 1. 适度原则
- 管控点不宜过多，避免影响效率
- 聚焦高风险、高价值的节点
- 低风险操作可采用事后审计

### 2. 可追溯原则
- 管控操作必须留有记录
- 支持事后追溯和审计
- 记录操作人、时间、内容

### 3. 可量化原则
- 管控条件尽量可量化
- 避免模糊的主观判断
- 便于系统自动化处理

### 4. 分级原则
- 根据风险等级设置不同管控强度
- 高风险：多级审批、实时监控
- 中风险：单级审批、定期检查
- 低风险：事后抽查、日志记录

## 管控点与系统设计的关系

管控点直接影响系统设计：

| 管控点类型 | 系统设计要求 |
|------------|--------------|
| 审批点 | 工作流引擎、审批配置、移动审批 |
| 检查点 | 规则引擎、校验服务、异常处理 |
| 监控点 | 监控系统、告警服务、仪表盘 |
| 记录点 | 日志系统、审计服务、归档存储 |
