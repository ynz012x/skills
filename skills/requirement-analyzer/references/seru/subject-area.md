# 主题域划分（SERU-S）

主题域（Subject Area）是按业务职责划分的系统子领域，用于明确系统边界和组织需求。

## 核心概念

**主题域的特征**：
- 具有内聚的业务职责
- 边界清晰，与其他主题域低耦合
- 包含一组相关的业务实体和用例
- 通常对应一个业务部门或职能领域

## 划分策略

### 策略1：按组织结构划分

适用于组织职能明确的系统。

**示例**（企业管理系统）：
- 人力资源域：员工、考勤、薪酬
- 财务域：预算、报销、账务
- 销售域：客户、订单、合同
- 采购域：供应商、采购单、入库

**优点**：与组织架构对应，职责清晰
**缺点**：可能导致跨部门流程的割裂

### 策略2：按业务流程划分

适用于流程导向的系统。

**示例**（电商系统）：
- 商品域：商品、分类、库存
- 交易域：购物车、订单、支付
- 履约域：发货、物流、签收
- 售后域：退款、退货、投诉

**优点**：符合业务流程自然边界
**缺点**：需要明确流程间的衔接点

### 策略3：按数据主题划分

适用于数据密集型系统。

**示例**（数据平台）：
- 用户域：用户画像、行为数据
- 交易域：订单数据、支付数据
- 内容域：商品数据、评价数据
- 分析域：报表、指标、洞察

**优点**：数据边界清晰，便于数据治理
**缺点**：可能忽视业务流程的完整性

### 策略4：混合划分

结合多种策略，根据业务特点灵活选择。

**原则**：
- 核心业务按流程划分
- 支撑功能按职能划分
- 数据服务独立成域

## 划分步骤

1. **识别业务领域**：列举系统涉及的所有业务领域
2. **分析职责边界**：确定每个领域的核心职责
3. **评估耦合度**：检查领域间的依赖关系
4. **合并或拆分**：
   - 职责重叠的领域考虑合并
   - 职责过多的领域考虑拆分
5. **定义接口**：明确领域间的数据交换和事件通信

## 主题域描述模板

```
主题域名称：XXX域

核心职责：
- 职责1
- 职责2
- 职责3

包含实体：
- 实体A
- 实体B

主要用例：
- 用例1
- 用例2

与其他主题域的关系：
- 向 YYY域 提供：XXX数据
- 从 ZZZ域 获取：ZZZ事件
```

## 常见陷阱

| 陷阱 | 表现 | 解决方案 |
|------|------|----------|
| 划分过细 | 主题域超过10个，边界模糊 | 合并相关性高的领域 |
| 划分过粗 | 单个主题域职责过多 | 按子职能拆分 |
| 边界重叠 | 同一实体出现在多个域 | 明确数据归属，定义主数据 |
| 孤岛效应 | 主题域间缺少连接 | 梳理跨域流程，补充接口 |
